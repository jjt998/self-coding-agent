# 代码可读性规范

## 目的

本规范用于统一本项目在开发过程中的代码可读性要求。

核心原则只有两条：

1. 代码只要存在理解门槛，就补相应的白话中文注释。
2. 只要属于描述类内容，默认一律使用中文。

## 白话中文注释规范

### 什么时候必须补注释

以下情况应主动补充白话中文注释：

1. 新增、修改、重构的代码中，存在明显理解门槛的逻辑。
2. 状态较多、分支较绕、流程较长的代码。
3. 读者如果只看代码本身，较难快速理解“为什么这样写”的地方。
4. 占位实现、过渡实现、兼容性写法、临时约束等容易让人误解的地方。

### 注释应该怎么写

注释要优先解释：

1. 这段代码在解决什么问题。
2. 为什么这里要这样写。
3. 这段逻辑和当前阶段目标之间的关系。

函数、方法、类如果对外承担明确职责，默认还应补三引号文档字符串，例如 `"""..."""` 或 `'''...'''`，优先说明：

1. 这个接口是做什么的。
2. 它处理的对象或输入输出大致是什么。
3. 它在当前模块里的职责边界是什么。

推荐写法示例：

- “这里先落一个占位文件，保证后续 run 目录结构稳定。”
- “这里提前写状态迁移事件，是为了后面接入状态机时不再改 trace 结构。”
- “这里把路径规范成绝对路径，避免从不同目录启动时 trace 内容不一致。”
- `"""构建命令行参数解析器，统一管理本次 run 的 CLI 输入。"""`
- `'''执行最小状态机链路，并返回最终运行态。'''`

### 什么样的注释不要写

避免这类没有信息增量的注释：

- “给变量赋值”
- “调用函数”
- “进入 if 判断”
- “返回结果”

### 注释取舍原则

1. 不要求每一行都写注释。
2. 能靠清晰命名解决的问题，优先靠命名解决。
3. 只要逻辑稍复杂，就优先补一条能帮助人快速读懂的中文注释。
4. 当“简洁”和“可读”冲突时，优先保证可读。
5. 函数、方法、类如果职责不够一眼看懂，应优先补三引号文档字符串，而不是只留零散行内注释。

## 描述类内容中文规范

### 默认要求

凡是描述类内容，默认一律使用中文。

描述类内容包括但不限于：

1. 代码注释
2. CLI 的 `description`
3. CLI 参数的 `help`
4. 用户可见的提示语
5. 报告标题与报告说明
6. 文档说明
7. 阶段进展说明
8. 设计说明
9. 报错分析说明
10. 提交说明草稿

### 必须保留英文的场景

以下内容可以保留英文原文：

1. 代码标识符
2. 协议字段
3. 第三方接口字段
4. 标准库名
5. 命令名
6. 文件格式名
7. 已经固定的 schema 枚举值

### 开发中的执行要求

1. 写新代码时，同时检查描述类字符串是否应改为中文。
2. 修改旧代码时，如果不影响外部兼容性，应优先把描述类英文改成中文。
3. 如果字符串既承担程序语义又承担展示说明，需要先判断它是不是外部契约字段；不是的话优先中文。

## 与项目流程的关系

1. 本规范适用于本项目所有后续开发阶段。
2. 涉及开发规范、代码风格、注释策略、描述语言选择时，应优先阅读本文件。
3. 若本文件与临时习惯冲突，以本文件为准。
